GenBundle is a tool for generating bundle classes and families of bundles from template bundle classes. The templates consider the resource bundles are stored separated from the bundle class it self. It assumes you can implement a specific {@link com.soluzionasf.ocen.util.genbundle.Device Device} (for example: {@link com.soluzionasf.ocen.util.genbundle.FileDevice FileDevice} or database device) for getting the resources. Setting command line arguments of the generators. The user can set the package, parent class, parent bundle, header class comment, the loader used for getting the resources, etc.

You can use two generators:
  • {@link com.soluzionasf.ocen.util.genbundle.GenBundle#main(java.lang.String[]) GenBundle.main(String)} lets you generate a bundle family (that is a base bundle class (without suffixes) and all of it possible locale bundle classes (with suffixes)) The user can customise via input argument options the generation process.
  • {@link com.soluzionasf.ocen.util.genbundle.GenTreeBundle#main(java.lang.String[]) GenTreeBundle.main(String)} This generator provides you a way for generating a set of bundle families, that is a set of base bundles and all its locales. It simply invokes successively the {@link com.soluzionasf.ocen.util.genbundle.GenBundle#main(java.lang.String[]) GenBundle.main(String)}. The input argument is a property file name where are collected all the necessary inputs.

The generated bundles have the following characteristics:

  • The resource bundles are stored independently from the bundle classes. They are stored in a efficient way (using static instances).
  • You can implement the way you want to access to the resource bundles. We assume they are stored on a certain device. If you want to implement another device for loading the resources you have create another implementation of the interface {@link com.soluzionasf.ocen.util.genbundle.Device Device}. We have provided an implementation for a file device: {@link com.soluzionasf.ocen.util.genbundle.FileDevice FileDevice}. But you can implement for example a database device.
  • The loading process is invoked on the creation of new bundle class instance, so, the resource contents have to be loading dynamically. On the generation process you can control witch class do you want to use for loading the resources dynamically. By default we provide the class {@link com.soluzionasf.ocen.util.genbundle.DeviceFactory DeviceFactory}, but you can define another specific one. For example you can define a wrapper class for {@link com.soluzionasf.ocen.util.genbundle.DeviceFactory DeviceFactory} that simply set the localisation of the property file where is stored the dynamic instance information.
  • You can specify the parent bundle hierarchy, in a way to generate a bundle tree dependence. So on the parent bundle you have bundle keys that you don't need to redefine them again. Such tree dependence is useful for working on big projects, where each module will have its own bundles, but all of them will share some common resource bundles for each language bundle specification.
