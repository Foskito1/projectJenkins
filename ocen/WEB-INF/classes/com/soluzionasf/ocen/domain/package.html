Common classes for all modules, associated with the specific business.

Some classes common to all modules, for example: {@link com.soluzionasf.ocen.domain.StandardService StandardService}, assumes the existence of an application XML properties file. Such file can be loaded using the class {@link isf.util.PropertiesLoader PropertiesLoader}, that assumes the existence of a properties manager file: PropertiesManager.properties. This file should define which files are loaded by {@link isf.util.PropertiesLoader PropertiesLoader}. For more information see the class {@link com.soluzionasf.ocen.util.Properties_helper Properties_helper} which is used for getting the properties values and node from the XML application properties files.

We have assumed there a application file that defines nodes in the way are defined on some the node classes used on the package.

The module classes and CenModStandardService, where Mod={Bdi, Cmb, Man, Ope}, are identical unless MOD, Mod, where replaced on source files by the corresponding acronym. This substitution was done using the following perl script: generateModClass.pl. The design of the classes tries to simplify the differences on the module classes, in the way that they delegates on parent classes or internal classes for getting the specific information for the given module class.

On the class {@link com.soluzionasf.ocen.domain.StandardService StandardService} there is documented how to proceed for creating the services using the framework of this package.
